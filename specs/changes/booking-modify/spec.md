# 規範：預約修改功能

## 1. 使用者故事 (User Stories)
- **作為顧客**，我希望能修改已建立的預約時間，這樣當我有突發行程時，不需要取消再重新預約，省去重複填寫資料的麻煩。
- **作為顧客**，我希望系統能告訴我哪些時段是可以改期的，避免我選到已被他人佔用的時間。

## 2. 驗收標準 (Acceptance Criteria)

### 場景：顯示修改按鈕 ✅
- **Given**：使用者已登入並位於「我的預約」頁面。
- **When**：預約狀態為 `pending` (待確認) 或 `confirmed` (已確認)，且距離預約時間大於 24 小時。
- **Then**：該預約項目應顯示「修改預約」按鈕。

### 場景：進行改期流程 ✅
- **Given**：使用者點擊了某個預約的「修改預約」按鈕。
- **When**：彈出改期視窗，顯示目前的服務名稱與預約時間。
- **Then**：使用者可以重新選擇日期與可選的時段。

### 場景：確認改期成功 ✅
- **Given**：使用者選好了新的日期與時段並點擊「確認修改」。
- **When**：系統檢查該時段未被佔用，且符合 24 小時限制。
- **Then**：預約記錄更新成功，舊預約時間被釋放，列表顯示新的時間，並顯示「修改成功」提示。

### 場景：違反 24 小時限制 ✅
- **Given**：預約時間在 24 小時內。
- **When**：使用者查看該預約項目。
- **Then**：不應顯示「修改預約」按鈕，或按鈕呈現禁用狀態並顯示「已過改期期限」。

## 3. 邊界情況 (Edge Cases)
- **時段競爭**：兩人在同一秒修改至同一時段（需透過資料庫約束或事務處理）。
- **網路中斷**：修改過程中斷應保持原預約狀態。
- **跨日修改**：若修改至跨日的時段，需確保日期與時間計算正確。

## 4. 資料模型與 RLS 變更
- **`public.bookings`**：更新 `scheduled_at` 欄位。
- **`public.time_slots` (新建立)**：
  - `id`: uuid
  - `service_id`: references services
  - `slot_time`: timestamp (或 start_time)
  - `is_booked`: boolean
- **RLS 策略**：
  ```sql
  -- 只有擁有者可以 update 自己的預約
  CREATE POLICY "Users can modify own bookings" ON public.bookings
  FOR UPDATE USING ( auth.uid() = user_id );
  ```
